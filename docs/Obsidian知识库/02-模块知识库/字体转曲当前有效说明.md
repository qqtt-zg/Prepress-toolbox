# 字体兼容与转曲当前有效说明

## 当前有效定义
`字体兼容与转曲` 当前应被理解为一条字体风险修复工作流，核心目标是降低字体缺失、替换和输出不一致问题。

字体转曲是其中的关键处理能力，但当前行为边界已明确：
- 转曲本身不再承担"自动完成最终保存"的职责
- 处理结果以内存形式回传给上层界面
- 由用户显式决定何时保存

## 为什么这样定义

现有资料中最明显的冲突是：
- 较早说明（`docs/PDF字体转曲_使用说明.md`）把转曲写成"执行后直接保存"
- 后续优化说明（`docs/PDF检查器_v1.3.5_转曲工作流优化.md`）已经明确改为"处理结果以内存形式返回，由父级界面统一接管保存"

因此当前有效定义必须以新的交互链路为准。

## 当前有效工作流
1. 从上层界面（如 [[PDF检查器]]）发起字体转曲
2. 底层转曲流程生成新的 PDF 结果
3. 结果通过完成事件回传给上层
4. 上层界面接收新 PDF 内容
5. 用户再决定保存、另存为或继续查看

## 当前关键机制

新的行为边界已经非常明确：
- 转曲完成后会触发完成事件（`FontOutlineCompleted`）
- 回传内容包含新的 PDF 数据与原始文件路径
- 保存责任从"底层转曲流程"上移到了"父级界面/上层工作流"

所以这里的关键不是"转曲工具怎么立即导出"，而是：
- **如何得到转曲后的结果**
- **如何在上层界面继续操作这个结果**

## 当前已明确可用
- 针对字体兼容问题执行转曲
- 将转曲结果回传到上层界面
- 支持用户后续手动保存

## 当前应判定为历史描述的内容

以下表述应视为旧工作流描述，不宜继续当作当前说明：
- "转曲成功后自动进入保存步骤"
- "转曲窗口自身负责完成最终文件输出"

## 与其他主题的关系
- 操作入口见 [[PDF检查器]]
- 与 [[工作台与排版工作区]] 的关系：都属于 PDF 工作流环节，但字体转曲偏修复，拼版偏输出
- 外部依赖与实现资料可归并到 Ghostscript、Poppler 等说明下

## 相关代码与外部依赖
- Ghostscript（转曲核心引擎）
- Poppler（字体检测辅助）
- PDF 检查器相关面板/服务
