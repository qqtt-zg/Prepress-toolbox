# 修复记录：主题预览表格表头被遮挡问题

## 问题描述

在 ThemeEditorControl 的主题预览区域，表格的表头被数据行内容遮挡，导致用户无法看到列标题。

### 问题表现
- 表头完全被数据行覆盖
- 滚动表格时，数据内容会覆盖表头区域
- 用户无法识别各列的含义
- 预览效果严重失真

## 问题原因

原始实现中，表头和数据区域的布局设计存在缺陷：

1. **错误的容器结构**：
   ```csharp
   tableContainer (Padding: 0, 28, 0, 0)
     ├─ dataArea (Dock.Fill, AutoScroll)
     └─ tableHeader (Anchor, 悬浮在 Padding 区域)
   ```

2. **Z-Order 问题**：虽然调用了 `tableHeader.BringToFront()`，但由于 `dataArea` 使用了 `AutoScroll`，滚动时内容会溢出并覆盖表头

3. **布局冲突**：`dataArea` 的 `Dock.Fill` 会填充整个容器（包括 Padding 区域），导致表头被遮挡

## 解决方案

重新设计布局结构，将表头从 `tableContainer` 中独立出来，直接添加到 `pnlPreview`：

### 新的布局结构
```
pnlPreview
  ├─ toolbar (Dock.Top, Height: 40)
  ├─ tableHeader (Dock.Top, Height: 28) ← 独立的表头
  ├─ tableContainer (Dock.Fill, AutoScroll) ← 仅包含数据行
  └─ statusBar (Dock.Bottom, Height: 20)
```

### 关键改动

1. **表头独立**：
   ```csharp
   var tableHeader = new Panel
   {
       Height = 28,
       Dock = DockStyle.Top,  // 使用 Dock.Top 而不是 Anchor
       BackColor = _editingTheme.SurfaceLight,
   };
   ```

2. **简化数据容器**：
   ```csharp
   var tableContainer = new Panel
   {
       Dock = DockStyle.Fill,
       BackColor = _editingTheme.Surface,
       AutoScroll = true
       // 移除 Padding，不再需要为表头预留空间
   };
   ```

3. **正确的添加顺序**：
   ```csharp
   pnlPreview.Controls.Add(statusBar);      // Bottom
   pnlPreview.Controls.Add(toolbar);        // Top
   pnlPreview.Controls.Add(tableContainer); // Fill
   pnlPreview.Controls.Add(tableHeader);    // Top (在 toolbar 之后)
   tableHeader.BringToFront();              // 确保在最顶层
   ```

4. **动态列宽计算**：
   ```csharp
   int availableWidth = pnlPreview.ClientSize.Width - 2;
   var headers = new[] 
   { 
       ("序号", (int)(availableWidth * 0.10)),
       ("原文件名", (int)(availableWidth * 0.25)),
       ("新文件名", (int)(availableWidth * 0.25)),
       ("订单号", (int)(availableWidth * 0.20)),
       ("材料", (int)(availableWidth * 0.20))
   };
   ```

### 修改文件
- `src/WindowsFormsApp3/Forms/Controls/Settings/ThemeEditorControl.cs`

## 技术说明

### Dock 布局的 Z-Order
- 使用 `Dock` 的控件按添加顺序堆叠
- `Dock.Top` 的控件从上到下堆叠
- `Dock.Bottom` 的控件从下到上堆叠
- `Dock.Fill` 填充剩余空间

### 为什么 Anchor 方案失败
1. **Anchor 的局限性**：`Anchor` 只能调整控件的位置和大小，不能改变 Z-Order
2. **AutoScroll 的影响**：启用 `AutoScroll` 的容器，其内容可以超出可见区域
3. **Padding 的问题**：`Dock.Fill` 会忽略父容器的 Padding，导致内容溢出

### 为什么新方案有效
1. **独立层级**：表头和数据区域在不同的层级，不会相互覆盖
2. **固定位置**：表头使用 `Dock.Top` 固定在顶部，不受滚动影响
3. **清晰的布局**：每个区域职责明确，易于维护
