# Action List 架构方向

## 概念来源
Action List（动作列表）概念源自 **Enfocus PitStop Pro**，是其最核心的自动化能力之一：将多个独立操作串联成可复用的处理流程，批量应用到 PDF 文件。

## 与本项目的关系
Action List 在本项目中不应被理解为"一个 UI 按钮或面板功能"，而应理解为**长期架构能力**——它影响的是整个服务层的组织方式。

当前项目中 Action List 的定位：
- 它是 [[PDF检查器]] 和 [[工作台与排版工作区]] 共享的底层自动化框架
- 它是 [[PDF拆分模块]] 未来扩展的架构基础
- 它不是短期 UI 修补目标

## 当前项目中的 Action List 设计

### 核心概念（来自 `docs/ActionList使用指南.md`）

**动作（Action）**
- 每个动作是一个独立的功能模块
- 例如：设置材料类型、计算平张布局、添加空白页

**动作列表（Action List）**
- 多个动作按顺序组合，形成完整的处理流程

**上下文（Context）**
- 动作之间通过上下文传递数据
- 前一个动作的输出可以作为后一个动作的输入

### 当前动作分类

| 类别 | 示例动作 |
|------|---------|
| 基础设置 | 设置材料类型、设置工艺参数、设置尺寸、设置形状 |
| 排版布局 | 设置材料规格、设置边距、设置行列、设置联数 |
| 布局计算 | 计算平张布局、计算卷装布局、应用旋转 |
| 空白页处理 | 计算空白页、添加空白页 |

### 当前使用场景
1. PDF 拆分流程中的布局计算
2. 批量处理任务
3. 动作列表配置的保存与加载

## 为什么 Action List 是长期架构

### 它影响服务层组织
当前项目的服务层（`ImpositionService`、`SaddleStitchService`、`PdfSplitService` 等）各自独立。Action List 架构要求：
- 每个服务方法都可以被封装为一个"动作"
- 动作之间通过标准化上下文传递数据
- 整个处理流程可以被录制、保存、回放

### 它连接多个模块
- [[工作台与排版工作区]]：拼版流程可以被表达为动作列表
- [[PDF检查器]]：检查与修复流程可以被表达为动作列表
- [[PDF拆分模块]]：拆分后的批量处理可以被表达为动作列表
- [[设置中心]]：动作列表配置可以被持久化

### 它是自动化的前提
热文件夹、批量处理、模板复用等高级功能，都依赖于"处理流程可被序列化和回放"这一前提。Action List 就是这个前提的架构实现。

## PitStop 的 Action List 参考

PitStop 的 Action List 系统支持：
- 录制手动操作
- 保存为可复用配置
- 批量应用到多个文件
- 条件执行（if/then 分支）
- 错误处理与日志

本项目当前只实现了 Action List 的基础框架（动作注册、上下文传递、顺序执行），条件执行和录制能力尚未实现。
